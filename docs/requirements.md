# 要件定義

対応する Rules は `docs/rules/requirements.md` を参照する。

## 1. 機能要件

### 1.1 入力

- アプリは画像ファイルを 1 枚以上受け付ける
- 対応する読み込み方法はファイル選択ダイアログのみとする
- ドラッグアンドドロップは受け付けない
- 複数ファイルが渡された場合は、受け付けた順にキューへ積んで順次処理する
- 空入力はエラーにせず無視する
- 起動時のコマンドライン引数は受け付けない

### 1.2 表示

- キューに積まれた入力ファイル名一覧を UI のテキストボックスへ表示する
- 現在処理中の対象と進行状況を UI の状態表示へ表示する
- 現在の入力状態をユーザーがすぐ分かるようにする
- 処理中は状態表示を行う
- 処理完了後は出力結果を確認できるようにする

### 1.3 変換

- AI を使って 2 倍、3 倍、4 倍拡大を行う
- デノイズは `0 / 1 / 2 / 3` から選択できるようにする
- 倍率は UI から選択できるようにする
- GPU を使って推論する
- 処理中に同じジョブを二重実行できないようにする
- キューに積まれたジョブは 1 件ずつ順に実行する

### 1.4 保存

- 出力画像は入力画像と同じフォルダへ自動保存する
- 出力形式は入力画像の拡張子を維持する
- 出力ファイル名は `{stem}-denoiseXx-upYx{suffix}` とする
- 初期の保存形式は PNG、JPEG / JPG、WebP を対象にする
- JPEG 保存時に入力画像または出力画像が透過を持つ場合は、白背景で合成して保存する

## 2. 非機能要件

### 2.1 操作性

- デモっぽい画面ではなく、常用できるデスクトップアプリの見た目にする
- メイン操作は数ステップで完了するようにする
- 不正な操作は UI 側で抑止する
- キューの進行状況と現在処理中の対象が分かるようにする

### 2.2 応答性

- 推論中に UI スレッドを止めない
- 処理開始、処理中、処理完了、失敗の状態が分かるようにする

### 2.3 拡張性

- 複数画像キューを無理なく扱える構造にする
- 今は不要な抽象化は増やしすぎない
- ただし UI と推論ロジックは分離する

### 2.4 保守性

- 画像読み込み経路を 1 本化する
- 推論実行経路を 1 本化する
- 単一画像処理と複数画像キュー処理で、1 ジョブ単位の実行ロジックを共有する
- モデルの差し替えが局所的に済むようにする

## 3. 入出力要件

### 3.1 初期対応入力形式

- PNG
- JPEG / JPG
- WebP

### 3.2 初期対応出力形式

- PNG
- JPEG / JPG
- WebP

## 4. エラー時の扱い

- 非対応形式を読み込んだ場合は短く分かりやすいエラーを表示する
- GPU 推論に失敗した場合は処理失敗を明示する
- 保存に失敗した場合は原因を推測しやすい文言にする
- キュー内の 1 件が失敗しても、その失敗を表示した上で残りのジョブ処理は継続できるようにする

## 5. 現時点で範囲外とする項目

- 複数画像の並列処理
- モデル切り替え UI
- タイル分割 UI
- 詳細画質パラメータ調整
- バックグラウンドの自動監視
- 履歴管理

## 6. 受け入れ条件

以下を満たしたら MVP の要件を満たしたと判断する。

1. 画像を 1 枚選択して入力一覧へ表示できる
2. 複数画像をファイル選択した場合に、キューへ積まれて順次処理される
3. 選択されたファイル名一覧が UI で確認できる
4. 2 倍、3 倍、4 倍の AI 拡大を GPU で実行できる
5. デノイズ `0 / 1 / 2 / 3` を指定して実行できる
6. 出力が `{stem}-denoiseXx-upYx{suffix}` で入力ファイルと共存できる
7. 透過画像を JPEG で保存した場合に白背景で合成される
8. 処理中に UI が固まらない
