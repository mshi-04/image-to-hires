# サブエージェント運用ガイド

## 1. 目的

この文書は、このリポジトリでサブエージェントを積極的に使うための基準を定めるものです。

主目的は次の 3 つです。

- 並列化できる作業を早く進める
- 主担当エージェントの認知負荷を下げる
- 調査、実装、検証を分担して進行速度を上げる

対象にする仕事は次の通りです。

- 調査
- 設計補助
- 実装補助
- 検証補助
- 文書整理

最終判断、最終統合、ユーザーへの応答責任は主担当エージェントが持つ。

## 2. 基本方針

- サブエージェントは積極的に使う
- ただし、分担すると遅くなる作業は主担当がそのまま処理する
- すぐ次の一手が止まる作業は、主担当が自分で処理することを優先する
- サブエージェントには、独立性が高く、境界が明確で、成果物が定義しやすい仕事を渡す

## 3. 使うべき場面

次のような場面では、サブエージェントの利用を第一候補として検討する。

- 複数の調査を並列に進められるとき
- UI と推論のように責務がきれいに分かれているとき
- 複数ファイルにまたがるが、担当範囲を分離しやすいとき
- 実装と検証を並列に進められるとき
- 一部の方針確認を進めつつ、主担当が別の作業を進められるとき

## 4. 使わない方がよい場面

次のような場面では、サブエージェントを使わないか、使っても最小限にする。

- すぐ次の操作がその結果待ちで止まるとき
- 小さすぎる修正で、委譲コストの方が高いとき
- 仕様が曖昧で、主担当の判断を先に固める必要があるとき
- 同じファイルを複数エージェントが同時に触りやすいとき

## 5. 推奨する役割分担

### 調査担当

- 既存コードや文書の確認
- 技術候補の比較
- 特定機能の影響範囲の洗い出し

### 実装担当

- 責務が分かれた範囲のコード追加
- UI 側の個別機能
- 推論側の個別機能
- テストや補助コードの追加

### 検証担当

- ビルドや起動確認
- テスト実行
- エラー再現手順の確認
- 変更点のざっとしたリスク確認

## 6. 分担の切り方

- 書き込み範囲が重ならないように切る
- 同一ファイルを複数のサブエージェントに持たせない
- 責務単位で ownership を明示する
- 依頼時に対象ファイル、目的、期待する出力を具体的に渡す

よい分担例:

- UI のドラッグアンドドロップ実装
- モデルローダーの実装
- キュー制御ロジックの実装
- 設計文書の更新案

悪い分担例:

- 同じメインウィンドウを 2 つのサブエージェントに同時編集させる
- 主担当自身も進めている箇所を重複して調査させる
- 依頼範囲が広すぎて成果物が曖昧なまま投げる

## 7. 依頼時に含めるべき情報

サブエージェントへ渡す依頼には、最低限次を含める。

- 何をしてほしいか
- なぜ必要か
- どのファイル、またはどの責務を担当するか
- 変更してよい範囲
- 期待する成果物
- 他のエージェントも並行作業している前提

可能なら次も含める。

- 前提条件
- 見てよい範囲
- 触ってよい範囲
- 優先度
- 必要な期限感

コード変更を依頼する場合は、次も含める。

- 自分以外の変更を消さないこと
- 競合しそうなら現状に合わせて調整すること
- 最終的に変更したファイル一覧を返すこと

依頼時の出力形式は、次のように短く構造化させるのが望ましい。

- 結論
- 根拠
- 未確定事項

## 8. 待機と進行のルール

- サブエージェントを出したら、主担当はすぐに別の非重複作業へ進む
- reflex 的に待たない
- すぐ必要な結果で本当に止まっているときだけ待つ
- 同じサブエージェントに対する無意味な追いメッセージは避ける

## 9. 結果統合のルール

- サブエージェントの成果を受け取ったら、主担当が最終責任を持って確認する
- 主担当は内容を鵜呑みにせず、変更範囲と前提を短く確認する
- 必要なら主担当が軽く整形、統合、追加修正を行う
- 最終的な説明は主担当が一貫した形でまとめる
- 統合時は、採用、不採用、保留を意識して整理する
- 根拠が弱い提案は、そのまま採用せず追加確認する

## 10. 競合回避ルール

- 同じファイルを複数のサブエージェントに編集させない
- 大きいファイルを触る場合は担当境界を先に決める
- 主担当が編集中のファイルを安易に委譲しない
- 競合しやすい場合は、調査だけ委譲して実装は主担当が行う

## 11. このリポジトリでの推奨パターン

### パターン 1: 調査を並列化する

- 主担当が実装方針を固める
- サブエージェント A がコードベースの関連箇所を調査する
- サブエージェント B が技術候補や差分影響を調査する
- 主担当は待たずに土台実装を進める

### パターン 2: UI と推論を分離する

- 主担当が共通インターフェースを定義する
- サブエージェント A が UI 側を担当する
- サブエージェント B が engine または pipeline 側を担当する
- 主担当が最終統合を行う

### パターン 3: 実装と検証を並列化する

- 主担当またはサブエージェントが実装する
- 別のサブエージェントが起動確認やテスト観点を確認する
- 主担当が必要な修正だけ回収する

## 12. 避けるべき運用

- 何も考えずに常にサブエージェントへ投げる
- 緊急のボトルネック作業まで委譲して止まる
- 同じ問題を複数エージェントで重複して解かせる
- 結果確認なしでそのまま採用する
- 文書やコードの前提がずれたまま放置する

## 13. 品質基準

- サブエージェントの出力は短くても検証可能であること
- 事実と推測を分けること
- できるだけ根拠となるファイルや箇所を示すこと
- 採否判断に使える粒度で返すこと

## 14. 文書更新ルール

サブエージェント運用の方針を変えたときは、この文書と `AGENTS.md` の文書マップを一緒に更新する。
