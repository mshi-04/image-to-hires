# UnitTest 方針

## Rules

- UnitTest は契約、分岐、状態遷移を守るために書く
- GPU 実機依存は最小化し、通常は CPU で回る境界テストを優先する
- 振る舞いを変えたら対応する正常系または失敗系のテストを追加する

## 1. 目的

この文書は、このリポジトリで UnitTest をどう書くかを定めるものです。

目的は次の通りです。

- ロジックの退行を早く検出する
- UI と推論の境界が崩れていないことを確認しやすくする
- 将来の複数画像対応やモデル差し替えで壊れやすい箇所を守る
- GPU がなくても回せるテストを増やす

## 2. 対象範囲

この文書の対象は次の通りです。

- Python 3 系のアプリケーションコード
- PySide6 を使う UI 周辺のうち、ロジックとして切り出せる部分
- PyTorch を使う推論コードのうち、境界と前後処理
- 画像読み込み、保存、倍率分岐などの処理
- 補助関数、設定処理、エラー変換処理

前提として、このプロジェクトは単一画像処理、Windows、NVIDIA GPU を対象にしている。

## 3. UnitTest の基本方針

- UnitTest は小さく、速く、再現可能であること
- 1 つのテストでは 1 つの振る舞いを確認する
- UI の見た目そのものではなく、状態遷移とロジックを重点的に確認する
- GPU 必須の確認は最小限にし、できるだけ CPU でも流せる境界テストに寄せる
- 推論そのものの正しさを全部テストするのではなく、呼び出し契約と前後処理を守る

## 4. 何を UnitTest の対象にするか

優先して UnitTest する対象は次の通り。

- 入力画像パスの検証
- 画像読み込みの共通入口
- 2 倍、3 倍、4 倍の倍率分岐
- 前処理と後処理
- 保存形式の分岐
- エラーをユーザー向け状態へ変換する処理
- 実行中フラグや状態遷移の制御
- UseCase や service の分岐ロジック

状態遷移は、少なくとも次の流れを意識して確認する。

- `idle -> loaded -> processing -> completed`
- `idle -> loaded -> processing -> failed`

## 5. UnitTest の対象外にしやすいもの

次のものは UnitTest の主対象にしない。

- 実際の見た目やレイアウトの細かい確認
- PyTorch モデル自体の学習品質
- GPU 実機がないと意味を持たないベンチマーク
- 外部ライブラリ内部の動作

これらは必要に応じて統合テスト、手動確認、実機確認で扱う。

## 6. テストの分割方針

- `domain` や純粋関数に近い処理は、最も厚く UnitTest する
- `domain/usecase` や service は、モックを使って分岐と状態遷移を検証する
- `infrastructure` は薄く保ち、境界条件だけを確認する
- `ui` はロジックを外へ逃がし、残った最小限だけをテストする

## 7. モックとスタブの方針

- ファイルシステムアクセスは原則モックまたは一時ディレクトリを使う
- GPU 実行部分はモック化して、呼び出し引数と戻り値の扱いを確認する
- モデルローダーは、実重みファイルを毎回読む前提にしない
- 画像データは、できるだけ小さいテスト用データで扱う

モックしすぎてテストの価値がなくならないよう、境界でだけ置き換える。

## 8. GPU 依存コードの扱い

- GPU が必要なテストは少数に絞る
- 大半のテストは engine のインターフェースをモックして流す
- `device` 選択、推論モード、前後処理の契約を確認する
- GPU 実機確認が必要なものは、UnitTest ではなく別枠で扱う

必要なら、GPU 依存テストには `gpu` や `slow` のような区別を付けて通常実行から分ける。

## 9. 画像データとテストフィクスチャ

- テスト用画像は小さく、再利用しやすいものを使う
- 透過 PNG、通常 JPEG、異常ファイルを最低限用意する
- 実行速度を落とす大きい画像は UnitTest に入れない
- バイナリ資産は増やしすぎない

## 10. 失敗系のテスト

最低限、次の失敗系を意識する。

- 非対応形式
- ファイル不存在
- 読み込み失敗
- 保存失敗
- モデル未配置
- GPU 利用不可
- 推論例外
- 不正な倍率指定

失敗時は、例外だけでなく状態遷移も確認する。

## 11. 命名規則

- テストファイル名は `test_*.py` にする
- テスト関数名は `test_<対象>_<条件>_<期待結果>` を基本にする
- テストクラスを使う場合も、責務単位でまとめる

よい例:

- `test_scale_factor_returns_expected_size`
- `test_load_input_image_rejects_unsupported_extension`
- `test_run_upscale_job_sets_busy_flag_during_execution`

## 12. ディレクトリ方針

テストは `tests/` 配下に置き、責務に対応させて分ける。

例:

```text
tests/
  unit/
    ui/
    domain/
    infrastructure/
  integration/
```

本体コードの責務分離に沿って、テストも同じ考え方で整理する。

## 13. テストデータの扱い

- テストデータは最小限に保つ
- 使い回せるフィクスチャを優先する
- 画像生成で済むものは、毎回ファイル資産を増やさず生成してよい
- 巨大なモデルや巨大画像をテスト資産として抱え込まない

## 14. 将来拡張を見据えた観点

- 単一画像前提でも、処理単位は job として扱えるようにテストを書く
- 将来の複数画像対応で再利用できる service を厚めにテストする
- 3 倍だけ別実装になる可能性を見据えて、倍率ごとの期待動作を個別に固定する
- モデル差し替え時も、`domain/usecase` 層の契約が変わらないことを重視する

## 15. レビュー時の確認点

- テストが実装詳細に依存しすぎていないか
- 成功系だけでなく失敗系も押さえているか
- GPU 必須テストを不必要に増やしていないか
- モックの境界が不自然ではないか
- 将来のリファクタで壊れにくい粒度になっているか

## 16. 維持運用ルール

- バグ修正時は、再発防止のテストを先に追加するか、同時に追加する
- 新しい分岐や例外を増やしたら、正常系と失敗系を最低 1 件ずつ追加する
- 重い、壊れやすい、環境依存の強いテストは UnitTest から外す
- 方針と実態がずれたら、この文書を更新する

## 17. 文書更新ルール

UnitTest の方針を変えた場合は、この文書と `AGENTS.md` の文書マップを一緒に更新する。
