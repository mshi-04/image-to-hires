# 技術設計

## 1. 技術選定

### 採用候補

- 言語: Python 3 系
- GUI: PySide6
- 推論: PyTorch + CUDA
- 画像入出力: Pillow
- 配列処理: NumPy

### ローカル確認済み環境

- 現在インストール済みの Python は `Python 3.13.13`
- このプロジェクトの手順や実行例は `python` コマンドを前提にする
- `py` ランチャーの利用可否は環境差があるため、このプロジェクトでは実行手順の基準にしない

### 採用理由

- PySide6 は Windows デスクトップアプリとして自然な GUI を作りやすい
- ファイルダイアログとの相性が良い
- Python は初期実装速度が高く、試行錯誤しやすい
- PyTorch はモデル読み込みや推論制御がしやすい
- 将来のモデル差し替えや検証がしやすい

## 2. 推論方式

### 基本方針

- 外部 CLI を叩く方式ではなく、アプリ内部で推論を実行する
- 推論エンジンを専用層に閉じ込め、UI から直接 PyTorch を触らない

### この方式を選ぶ理由

- 1 ジョブ単位の処理をアプリ全体で一貫して扱いやすい
- 複数画像キューの順次実行や状態管理へ拡張しやすい
- 入出力、失敗処理、進捗表示をアプリ側で一貫して管理できる

## 3. モデル利用方針

### 前提

- モデルは既存の学習済みモデルを使う
- 自前学習は MVP の対象外

### 有力候補

- Real-ESRGAN 系を中心に、2 倍、3 倍、4 倍へ対応できるモデル構成を検討する
- 3 倍専用モデルの候補が弱い場合は、3 倍だけ別方式を採る余地を残す

### 守るべき条件

- 他プロジェクトのソースコードを丸ごと取り込まない
- ライブラリとして利用できるなら依存として扱う
- 学習済み重みを使う場合はライセンスを確認する
- 必要最小限の統合コードだけをこのリポジトリに書く

## 4. GPU 前提

- NVIDIA GeForce GPU を前提とする
- CUDA が使える環境を前提とする
- CPU フォールバックは初期段階では重視しない

## 5. パフォーマンス上の注意

- 画像サイズが大きい場合、VRAM 使用量が増える
- モデルや入力サイズによっては推論時間が長くなる
- MVP ではまず安定動作を優先し、極端な最適化は後回しにする

## 6. 実行形態

### 基本方針

- 実装上の入口は `src/main.py` を維持する
- 利用者向けの起動導線は Windows の `.exe` を主にする
- `.exe` は `PyInstaller` の `onedir` 形式で作成し、Python ランタイムと依存を同梱する

### この方式を選ぶ理由

- ローカル環境に Python がない端末でも起動しやすい
- `onefile` よりトラブルシュートしやすく、開発中の起動検証が安定しやすい
- 推論処理は現状どおりアプリ内部実行を維持できる

## 7. リスクと対策

### リスク

- PyTorch と CUDA の組み合わせで環境差異が出る
- 大きい画像で VRAM 不足が起こる可能性がある
- モデルによって写真向けとイラスト向けの癖が出る

### 対策

- モデル読み込みを独立させる
- 推論処理を UI から分離する
- サイズ制限や事前チェックを入れやすい構造にする
- モデル差し替えを局所化する
