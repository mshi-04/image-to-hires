# 技術設計

## 1. 技術選定

### 採用候補

- 言語: Python 3 系
- GUI: PySide6 (Material風スタイル適用として qt-material のライトテーマを使用)
- 推論: Real-CUGAN (外部 CLI 呼び出し) + PyTorch + CUDA (将来的または一部処理)
- 画像入出力: Pillow
- 配列処理: NumPy

### ローカル確認済み環境

- 現在インストール済みの Python は `Python 3.13.13`
- このプロジェクトの手順や実行例は `python` コマンドを前提にする
- `py` ランチャーの利用可否は環境差があるため、このプロジェクトでは実行手順の基準にしない

### 採用理由

- PySide6 は Windows デスクトップアプリとして自然な GUI を作りやすい
- ファイルダイアログとの相性が良い
- Python は初期実装速度が高く、試行錯誤しやすい
- PyTorch はモデル読み込みや推論制御がしやすい
- 将来のモデル差し替えや検証がしやすい

## 2. 推論方式

### 基本方針

- 外部 CLI (Real-CUGAN ncnn-vulkan 等) をサブプロセスとして呼び出す方式を採用する
- 推論エンジンを専用層に閉じ込め、UI からは詳細な実行手順（プロセス呼び出しやオプション）を意識させない
- 2x / 3x / 4x は Real-CUGAN 経由で推論する

### この方式を選ぶ理由

- 外部 CLI を利用することで、推論時の VRAM 不足やクラッシュが UI プロセスに波及するのを防ぐことができる
- 環境依存になりやすい PyTorch 等の巨大な依存をアプリプロセス内に抱え込まず、配布やパッケージ化が容易になる
- 1 ジョブ単位の処理を抽象化インターフェースでラップすることで、要件である「複数画像キューの順次実行」は十分に達成できる

## 3. モデル利用方針

### 前提

- モデルは既存の学習済みモデルを使う
- 自前学習は MVP の対象外

### 有力候補

- Real-ESRGAN 系を中心に、2 倍、3 倍、4 倍へ対応できるモデル構成を検討する
- 3 倍専用モデルの候補が弱い場合は、3 倍だけ別方式を採る余地を残す

### 守るべき条件

- 他プロジェクトのソースコードを丸ごと取り込まない
- ライブラリとして利用できるなら依存として扱う
- 学習済み重みを使う場合はライセンスを確認する
- 必要最小限の統合コードだけをこのリポジトリに書く

## 4. GPU 前提

- NVIDIA GeForce GPU を前提とする
- CUDA が使える環境を前提とする
- CPU フォールバックは初期段階では重視しない

## 5. パフォーマンス上の注意

- 画像サイズが大きい場合、VRAM 使用量が増える
- モデルや入力サイズによっては推論時間が長くなる
- GPU 都合で複数画像の並列生成は採用しない
- 高速化は順次処理の範囲で行う
- Real-CUGAN の `-j` を明示し、小画像は `4:4:4`、大画像は `2:2:2` を使う
- 2x / 3x / 4x では、入力が `.png` / `.jpg` / `.jpeg` / `.webp` かつ EXIF 回転補正不要かつ色モード変換不要なら、元ファイルをそのまま Real-CUGAN へ渡す
- EXIF 回転補正や色モード変換が必要な場合だけ、Pillow で正規化した一時 PNG を Real-CUGAN へ渡す
- Real-CUGAN 実行時の一時ファイルはジョブごとにランダム名を作らず、エンジン単位の作業ディレクトリ内で固定名を再利用する
- 通常の実行では出力先フォルダ配下の `.tmp-realcugan-<engine-id>/` を使い、`input.png`、`realcugan.png`、`encoded.jpg` / `encoded.webp` / `encoded.png` を使い回す
- 出力先が未確定の経路だけ `tmp/realcugan-work/<engine-id>/` を使う
- 保存処理は原子保存を維持しつつ、不要な同期 I/O を減らしてオーバーヘッドを下げる
- 保存時の `fsync` は使わず、停電や強制終了時の耐久性よりも順次処理全体のスループットを優先する
- 出力形式は入力画像の拡張子を維持する
- 保存品質の既定値は PNG が可逆、JPEG が `quality=100` かつ `subsampling=0`、WebP が `lossless=True` かつ `quality=100` とする

## 6. 実行形態

### 基本方針

- 実装上の入口は `src/main.py` を維持する
- 利用者向けの起動導線は Windows の `.exe` を主にする
- `.exe` は `PyInstaller` の `onedir` 形式で作成し、Python ランタイムと依存を同梱する

### この方式を選ぶ理由

- ローカル環境に Python がない端末でも起動しやすい
- `onefile` よりトラブルシュートしやすく、開発中の起動検証が安定しやすい
- 推論処理は現状どおりアプリ内部実行を維持できる

## 7. リスクと対策

### リスク

- PyTorch と CUDA の組み合わせで環境差異が出る
- 大きい画像で VRAM 不足が起こる可能性がある
- モデルによって写真向けとイラスト向けの癖が出る

### 対策

- モデル読み込みを独立させる
- 推論処理を UI から分離する
- サイズ制限や事前チェックを入れやすい構造にする
- モデル差し替えを局所化する
