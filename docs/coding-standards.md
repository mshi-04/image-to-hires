# コーディング規約

対応する Rules は `docs/rules/coding-standards.md` を参照する。

## 1. 目的

この文書は、このリポジトリで実装を進めるときの基本的な書き方を定めるものです。

目的は次の通りです。

- 読みやすいコードを維持する
- UI と推論の責務分離を崩さない
- 将来の機能追加で破綻しにくい土台を保つ
- 主担当エージェントとサブエージェントの実装品質を揃える

## 2. 基本原則

- 小さく読める単位で書く
- 責務をまたぐ近道を避ける
- まず明確さを優先し、その後で必要な最適化を行う
- 今必要な拡張性だけを持たせ、過剰な抽象化を避ける
- 同じ意味の処理経路を複数作らない

## 3. 対象技術と前提

- 言語は Python 3 系を前提とする
- ローカルの確認済み実行環境は Python 3.13.13
- GUI は PySide6 を前提とする
- 推論は PyTorch + CUDA を前提とする
- 画像 I/O は Pillow を前提とする

技術選定が変わった場合は、この文書も更新する。

ローカル環境では `python` コマンドを使う前提で進める。

## 4. ディレクトリと責務

責務は `docs/architecture.md` に従って分ける。

- `ui/` は表示とユーザー操作を担当する
- `domain/` は値オブジェクト、ポート、ドメインサービス、状態遷移ルールを担当する
- `domain/usecase/` はアプリケーションのユースケースを担当する
- `infrastructure/` は推論、画像 I/O、モデルロード、永続化など外部依存を担当する

次のことを禁止する。

- UI から直接モデルをロードする
- widget 内で画像処理本体を完結させる
- 推論コードから widget の状態を直接変更する
- infrastructure から UI を参照する

## 5. 命名規則

### ファイル名

- Python ファイルは `snake_case.py` を使う
- 役割が分かる名前にする
- `utils.py` や `helper.py` のような曖昧な名前は避ける

### 変数名と関数名

- `snake_case` を使う
- 省略しすぎない
- 型ではなく意味で名前を付ける

よい例:

- `input_image_path`
- `load_input_image`
- `run_upscale_job`

避ける例:

- `img`
- `do_it`
- `data2`

### クラス名

- `PascalCase` を使う
- 役割を明示する

よい例:

- `MainWindow`
- `UpscaleWorker`
- `TorchUpscaleEngine`

## 6. 関数とクラスの設計

- 1 つの関数には 1 つの主要責務だけを持たせる
- 長い関数は意味のある単位で分割する
- クラスは状態を持つ理由があるときだけ作る
- 単なる処理の寄せ集めをクラス化しない
- 入口の検証ではガード節を積極的に使う

目安として、1 つの関数が長くなりすぎて画面 1 枚で追えないなら分割を検討する。

## 7. 値オブジェクト

- 不正値を閉じ込めたいものは値オブジェクトを使う
- バリデーション、等価性、不変条件を持つものを優先する
- 値オブジェクトは不変に保つ

候補例:

- `ScaleFactor`
- `InputImagePath`
- `OutputImagePath`
- `ImageSize`
- `ModelId`

単なる一時表示データまで値オブジェクト化しすぎない。

## 8. UI 実装ルール

- UI 層は入力受付、表示更新、イベント接続に集中する
- UI 層は推論の詳細を知らないようにする
- 重い処理は UI スレッドで動かさない
- ボタンの有効無効は状態に基づいて制御する
- エラーメッセージは短く、ユーザーに必要な内容だけ出す

UI コードに次のものを埋め込まない。

- モデル固有の前処理
- CUDA 判定ロジック
- 保存形式の詳細変換ロジック

UI 変更時の検証ルール:

- `src/ui/` 配下の Python ファイルを変更した場合は、コミット前に必ずコンパイル確認を行う
- コンパイル確認は `python -m compileall src/ui` を使う
- コンパイルエラーが 1 件でもある状態ではコミットしない

## 9. 推論と画像処理のルール

- 推論と保存の入口は `domain/usecase` に集約する
- 入力画像の読み込み、前処理、推論、後処理、保存の境界を明確にする
- 変換倍率の分岐は 1 か所に寄せる
- GPU 前提の処理と汎用画像処理を分ける
- モデル差し替え時に UI の修正が広がらない構造を保つ

## 10. 状態管理のルール

- 入力画像、出力画像、実行中フラグ、エラー状態を明示的に持つ
- 暗黙の状態遷移を避ける
- 複数の widget に同じ状態を分散保持しない
- 状態変更の起点を追いやすくする

## 11. 例外処理

- 例外は握りつぶさない
- 想定できる失敗は、意味のある単位で捕まえて扱う
- ユーザー向けメッセージと内部原因を混同しない
- 失敗時に中途半端な状態を残さない
- 異常系は深いネストではなくガード節で早めに返す

避ける例:

- `except Exception: pass`
- どこで失敗したか分からない汎用エラー表示

## 12. コメントと文書化

- コメントは必要なときだけ書く
- コードを見れば分かることはコメントしない
- 複雑な判断理由、制約、回避策はコメントしてよい
- 公開的な役割を持つ関数やクラスは、必要に応じて短い説明を添える

コメントは実装の言い換えではなく、意図や理由を書く。

## 13. インポートと依存関係

- 不要な依存を増やさない
- 依存関係の向きは `ui -> domain <- infrastructure` を意識する
- 下位層が上位層を参照しないようにする
- 循環参照を作らない

## 14. 設定値と定数

- パス、拡張子、既定値は散らさずに管理する
- マジックナンバーを直接埋め込まない
- ただし、過度な設定ファイル化はしない

## 15. ログとデバッグ出力

- 一時的な `print` を放置しない
- 必要なら後で差し替えやすい形で出力する
- ユーザー向け表示と開発向け情報を混ぜない

## 16. テスト観点

最低限、次の観点を意識して実装する。
- ファイル選択した複数画像が同じ読み込み経路へ入るか
- 複数ファイルがキューへ積まれて順次処理されるか
- 2 倍、3 倍、4 倍の分岐が正しく動くか
- デノイズ `-1 / 0 / 1 / 2 / 3`（`-1` はデノイズなし）の分岐と出力名が正しく動くか
- 推論中に UI が固まらないか
- 保存前後の状態遷移が崩れないか
- 異常系で UI が不整合にならないか

## 17. 将来拡張時の注意

- 1 ジョブ単位の処理を独立単位として保ち、キュー処理から再利用できるようにする
- 3 倍だけ特殊実装になる場合でも、分岐を散らさない
- タイル分割やモデル切り替えを入れるときも UI と推論の境界を守る
- 先回りしすぎた抽象化は入れない

## 18. レビュー観点

レビュー時は次を重点的に見る。

- 責務分離が崩れていないか
- 同じ意味の処理が複数経路に増えていないか
- エラー時の状態遷移が壊れていないか
- 一時実装が恒久化していないか
- キュー処理の導入で 1 ジョブ単位の処理が崩れていないか
- 値オブジェクトで閉じ込めるべき不変条件が漏れていないか
- ネストを減らせる箇所でガード節が使われているか

## 19. 文書更新ルール

この規約を変えた場合は、`AGENTS.md` の文書マップと必要な関連文書も一緒に更新する。
