# アーキテクチャ設計

## 1. 基本方針

最初から責務を分ける。複数ファイルを順次処理する段階でも、UI と推論を混ぜない。

複数画像キューを扱う場合でも、1 ジョブ単位の処理をそのまま順次流せる構造にする。

アーキテクチャの大分類は `ui / domain / infrastructure` とする。

ユースケースは `domain/usecase` に置く。

## 2. 想定ディレクトリ構成

```text
src/
  ui/
    windows/
    widgets/
  domain/
    entities/
    value_objects/
    usecase/
    ports/
    services/
  infrastructure/
    image_io/
    inference/
    model_store/
    persistence/
  main.py
```

## 3. 各層の責務

### ui

- メインウィンドウ
- ドラッグアンドドロップ
- ボタン操作
- プレビュー表示
- 状態表示

### domain

- エンティティ
- 値オブジェクト
- ユースケース
- ポート
- ドメインサービス
- アプリ固有の状態遷移ルール

### infrastructure

- 画像読み込みと保存
- PyTorch + CUDA を使った推論実装
- モデル重みの解決とロード
- 外部ライブラリやファイルシステムとの接続

## 4. データの流れ

1. UI がファイル選択またはドラッグアンドドロップで画像読み込み要求を出す
2. `domain/usecase` が入力を受けて検証し、1 件以上のジョブへ正規化する
3. 必要な値オブジェクトを組み立て、キューへ積む
4. usecase がキュー先頭のジョブを取り出して port を通して infrastructure を呼ぶ
5. infrastructure が画像 I/O や推論を実行する
6. usecase が結果をドメインの戻り値へまとめ、次ジョブの実行可否を判断する
7. UI がキュー状態と結果を表示し、保存要求を受ける

## 5. 設計ルール

- UI から直接 PyTorch、モデル、ファイル保存の詳細を触らない
- ファイル選択とドラッグアンドドロップは同じ読み込み経路へ流す
- 推論や保存の入口は usecase にまとめる
- 不変条件はできるだけ値オブジェクトへ閉じ込める
- 入力検証ではガード節を積極的に使う
- 画像 1 件の処理を独立したジョブ単位として扱い、複数ファイルはそのジョブを順次実行する

## 6. 将来拡張の見通し

複数画像キューの責務は、主に `domain/usecase` 側で管理する。

`infrastructure` は、1 ジョブ単位の処理をそのまま流用できる形を保つ。
