# レビューガイド

対応する Rules は `docs/rules/review-guide.md` を参照する。

## 1. 目的

この文書は、このリポジトリでレビューするときに何を優先して見るかを定めるものです。

目的は次の通りです。

- バグ、回帰、設計崩れを早く見つける
- `ui / domain / infrastructure` の責務分離を守る
- `domain/usecase` の入口が肥大化したり崩れたりするのを防ぐ
- GUI アプリと GPU 推論特有の事故を減らす

対象は、Python 3 系、PySide6、PyTorch、Clean Architecture 前提の実装、テスト、文書更新とする。

MVP 前提として、複数画像の順次処理、`2x / 3x / 4x`、Windows + NVIDIA GPU を対象にする。

## 2. レビューの基本姿勢

- まずバグと回帰リスクを探す
- 次に設計の崩れを探す
- その後に可読性や細部の改善を見る
- 好みだけの指摘は後順位にする
- 変更規模に対して過剰なリファクタ要求はしない

## 3. 優先順位

レビューでは次の順で確認する。

1. 動作破壊の有無
2. 例外時の挙動
3. 状態遷移の整合
4. アーキテクチャ違反の有無
5. テスト不足
6. 可読性
7. 命名や細かなスタイル

## 4. まず見るべき観点

### 4.1 仕様との整合

- 変更が現在の要件と矛盾していないか
- MVP の範囲を勝手に広げていないか
- 2x / 3x / 4x の扱いに漏れがないか

### 4.2 アーキテクチャ

- `ui` が推論詳細を直接持っていないか
- `domain/usecase` に処理の入口が集約されているか
- `infrastructure` から `ui` を参照していないか
- 外部依存の都合が `domain` に漏れていないか

### 4.3 値オブジェクトとガード節

- `ScaleFactor` など値オブジェクトで閉じ込めるべき不変条件が漏れていないか
- 深いネストを避け、ガード節で早めに弾ける箇所を放置していないか
- 入力検証が UI ごとに重複していないか

## 5. GUI アプリとしての確認点

- ファイル選択の経路で入力が正しく受け付けられているか
- 複数ファイルがキューへ積まれ、順次処理されるか
- UI スレッドで重い処理を実行していないか
- 実行中のボタン制御や状態表示が壊れていないか
- エラー時に UI が中途半端な状態で止まらないか
- 保存前後の状態遷移が整っているか

## 6. GPU / 推論まわりの確認点

- `torch` や CUDA の扱いが UI に漏れていないか
- GPU 利用可否やモデル未配置時の失敗処理があるか
- 前処理、推論、後処理、保存の境界が分かれているか
- 3x だけ別実装になる箇所で分岐が散っていないか
- モデル差し替え時に修正範囲が過剰に広がらないか
- `dtype`、推論モード、デバイス選択が暗黙になっていないか

## 7. UnitTest / Lint の確認

- 新しい分岐や例外に対してテストが追加されているか
- 正常系だけでなく失敗系も押さえているか
- GPU 必須テストを不必要に増やしていないか
- PR に UnitTest と Lint の実行結果が記載されているか
- 実行していない場合、その理由が書かれているか

## 8. 文書整合の確認

- 目的や MVP 範囲変更なら `docs/overview.md` が更新されているか
- 仕様変更なら `docs/requirements.md` が更新されているか
- 要件の判断基準変更なら `docs/rules/requirements.md` が更新されているか
- 技術方針変更なら `docs/technical-design.md` が更新されているか
- 構造変更なら `docs/architecture.md` が更新されているか
- 実装計画変更なら `docs/implementation-plan.md` が更新されているか
- セットアップ手順変更なら `docs/setup.md` が更新されているか
- サブエージェント運用変更なら `docs/subagent-guide.md` が更新されているか
- 実装ルール変更なら `docs/coding-standards.md` が更新されているか
- 実装ルールの判断基準変更なら `docs/rules/coding-standards.md` が更新されているか
- テスト方針変更なら `docs/unit-test-guidelines.md` が更新されているか
- テスト方針の判断基準変更なら `docs/rules/unit-test-guidelines.md` が更新されているか
- レビュー観点の判断基準変更なら `docs/rules/review-guide.md` が更新されているか
- Git 運用変更なら `docs/git-guidelines.md` と `docs/rules/git-guidelines.md` が更新されているか
- 入口や文書マップに影響があるなら `AGENTS.md` が更新されているか

## 9. 見落としやすい点

- 2x は直っているが 3x / 4x が壊れている
- 成功時だけ見て失敗時を確認していない
- UI の見た目は動くが状態遷移が壊れている
- 一部の入力経路だけ別ロジックになっている
- 空入力をエラーにすべき箇所と無視すべき箇所が混在している
- キュー先頭だけ成功し、後続ジョブの状態遷移が壊れている
- 値オブジェクトを通さずに生の値を横流ししている
- `domain/usecase` に外部ライブラリ依存が混ざっている
- RGB/BGR、PIL/Tensor、整数/浮動小数の変換でズレる
- 透過 PNG の扱いが漏れる
- 3x だけ保存サイズやプレビュー結果がずれる

## 10. コメントの書き方

- まず事実を書く
- 次に何が問題かを書く
- 最後にどう直すべきか、または何を確認すべきかを書く
- 好みの指摘とバグの指摘を混ぜない

## 11. レビュー結果の分類

レビュー結果は、最低限次の区分を意識する。

- `Critical`
- `Suggestion`
- `Nitpick`

レビューコメントを書くときは、各指摘をこの 3 区分のいずれかへ必ず分類する。

- `Critical` は、動作破壊、仕様逸脱、重大な設計違反、マージ前に修正が必要な問題に使う
- `Suggestion` は、改善した方がよい設計、保守性、テスト、可読性の指摘に使う
- `Nitpick` は、軽微な文言、命名、表現、スタイルの指摘に使う

レビュー結果を列挙するときは、上から `1.`, `2.`, `3.` のように項番を振る。

項番はレビュー本文の先頭から順に採番し、途中の区分変更で番号を振り直さない。

## 12. マージ判断

- 重大な不具合、仕様逸脱、設計違反がなければレビュー通過候補とする
- 軽微な命名や文言修正だけなら、別 PR に分けてもよい
- UnitTest や Lint を未実行にする場合は、理由が PR に残っていることを確認する
- 既知の妥協点や未解決事項は PR 本文に残す

## 13. 文書更新ルール

レビューの観点や優先順位を変えた場合は、この文書と `AGENTS.md` の文書マップを一緒に更新する。
